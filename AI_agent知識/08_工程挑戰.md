# 08 — 工程挑戰：Agent 上線會遇到什麼問題？怎麼解？

> **核心問題**：理論上 Agent 很強大，但真正上線時會撞上三道牆：不可靠、太貴、不可控。每道牆都有明確的工程解法。

---

## 三道牆總覽

```
關卡一：不可靠
  問題：幻覺、格式錯誤、無限迴圈
  解法：結構化輸出 + Guardrails + Verification

關卡二：太貴
  問題：Token 爆炸、每步都呼叫最強模型
  解法：Prompt Caching + 模型分級 + RAG 精準 Retrieval

關卡三：不可控
  問題：Agent 自己亂跑，做了不該做的事
  解法：Human-in-the-loop（高風險操作暫停確認）
```

---

## 關卡一：不可靠

### 問題一：幻覺（Hallucination）

模型在不確定時，傾向編造一個「看起來合理」的答案。

```
User: 台積電 2024 Q3 的確切毛利率是多少？
AI:   43.7%（但這是模型編的，不是真實數字）
```

**解法 A：工具接地（Tool Grounding）**
```
讓模型去查，不要靠記憶回答數字類問題：
→ Action: search("台積電 2024 Q3 毛利率")
→ 用真實搜尋結果回答，不猜測
```

**解法 B：不確定性表達**
```
在 Prompt 加入指令：
「如果你不確定某個數字，請明確說『我不確定，建議查閱原始財報』，
不要猜測或編造。」
```

---

### 問題二：格式錯誤

Agent 輸出的格式不固定，後續程式碼解析失敗。

```python
# ❌ 讓模型自由輸出
response = llm.invoke("分析這個合約的風險")
# 每次輸出格式不一樣，有時是 markdown，有時是純文字，有時是 JSON
# 後續程式碼 parser 時常失敗
```

**解法：結構化輸出（Structured Output）**

```python
from pydantic import BaseModel
from typing import List, Literal

class ContractRisk(BaseModel):
    risk_level: Literal["high", "medium", "low"]
    issues: List[dict]  # {"clause": "第3條", "problem": "...", "suggestion": "..."}
    confidence: float   # 0.0 ~ 1.0

# OpenAI / Anthropic 都支援強制輸出特定 schema
response = llm.with_structured_output(ContractRisk).invoke(prompt)
# 格式永遠正確，型別保證
```

---

### 問題三：無限迴圈

Agent 找不到答案，但也不知道怎麼停止，一直重複呼叫工具。

```
Thought: 我需要查詢 xxx
Action: search("xxx")
Observation: 沒找到相關結果

Thought: 我需要查詢 xxx
Action: search("xxx 相關")     ← 換個方式再試
Observation: 還是沒有

Thought: 我需要查詢 xxx
Action: search("xxx 詳細資料") ← 又試一次...
（無限下去）
```

**解法：最大迭代次數 + 明確停止條件**

```python
agent = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,          # 最多跑 10 次
    max_execution_time=60,      # 最多跑 60 秒
    early_stopping_method="generate",  # 超過限制時，強制生成最終答案
)
```

---

### 解法架構：Guardrails（護欄機制）

把驗證邏輯包在 Agent 的輸入和輸出兩端：

```
用戶輸入
   │
[Input Guardrail]
  檢查：
  - 是否包含 Prompt Injection？（嘗試操控 AI 的惡意指令）
  - 是否在允許的任務範圍內？
  - 是否包含敏感資訊？
   │
LLM 處理
   │
[Output Guardrail]
  檢查：
  - 輸出格式是否正確？
  - 是否包含幻覺（可驗證的事實錯誤）？
  - 是否提到不該說的內容？
   │
輸出給用戶

代表工具：NeMo Guardrails（NVIDIA）、Guardrails AI
```

---

### 解法：Verification Step（自我驗證）

```
Agent 產出答案
   ↓
再呼叫一次 LLM（可以用便宜的模型）：
「請檢查以下答案，有沒有邏輯錯誤、數字不一致、或明顯的幻覺？
 如果有問題，請指出。如果沒有，請回覆 'PASS'。」
   ↓
PASS → 輸出
有問題 → 重新生成（帶著問題描述）

成本：多一次 LLM 呼叫（用便宜模型可省費用）
收益：準確率顯著提升（尤其是數字、事實類任務）
```

## 關卡二：太貴

**費用公式**：
```
費用 = (輸入 token + 輸出 token) × 單價 × 呼叫次數
```

從三個方向降成本。

### 降成本策略一：Prompt Caching

System Prompt 通常很長（幾千 token），但每次呼叫都要重新傳。

```
沒有 Caching：
每次呼叫 = System Prompt（3000 token） + 對話（500 token）
→ 每次都付 3500 token

有 Prompt Caching（OpenAI / Anthropic 都支援）：
第一次：付完整 3500 token
之後：System Prompt 快取，只付 500 token 對話部分
→ 省 85% 的費用
```

### 降成本策略二：模型分級（最直接有效）

```
任務分級使用對應模型：

簡單分類、格式化、摘要
  → GPT-4o mini / Claude Haiku   （便宜 10~50 倍）

一般推理、問答、RAG
  → GPT-4o / Claude Sonnet       （中等）

複雜規劃、深度分析、Multi-Agent Orchestrator
  → o1 / Claude Opus             （貴但值得）

實際上 80% 的任務不需要最強的模型
```

### 降成本策略三：RAG 精準 Retrieval

```
❌ 把整份文件塞進 Prompt（100 頁 = 幾萬 token）

✅ 只傳相關的 3~5 個 chunk（幾百 token）
   靠 Reranking 確保找到的是真正相關的

→ 費用差距可達 100 倍以上
```

### 降成本策略四：Response Caching

```
同樣的問題 → 直接回傳上次的答案，完全不呼叫 LLM

實作：Redis + 問題的語意相似度判斷
適合：FAQ、固定查詢類、不需要即時性的回答

注意：需要考慮快取失效（文件更新後舊快取要清除）
```

## 關卡三：不可控

### 風險矩陣

```
              高風險操作
                  │
 確定性低  ───────┼─────── 確定性高
（不確定正確）     │        （很確定正確）
                  │
  ┌───────────────┼───────────────┐
  │               │               │
暫停詢問       需確認後執行    自動執行+記錄
（左上）        （右上）        （右下）
  │               │               │
報錯，請人處理   ？              自動執行
（左下）                          （右下）
              低風險操作
```

### 具體分類

```
✅ 自動執行（安全）：
  讀取資料、搜尋網路、查詢資料庫、生成草稿

⚠️ 需要人確認：
  發送 Email、修改資料庫、呼叫付費 API、費用超過 $X

❌ 絕對不自動執行：
  刪除資料、付款交易、公開發布、修改系統權限
```

### Human-in-the-loop 實作：Interrupt Pattern

```
Agent 執行中...
   │
   ▼
遇到「高風險動作」
（例如：即將發送 Email）
   │
   ▼
暫停，通知人類：
┌─────────────────────────────────┐
│ ⚠️  Agent 即將執行以下操作：     │
│                                 │
│  發送 Email 給 team@company.com │
│  主旨：Q3 財務報告               │
│  附件：report.pdf               │
│                                 │
│  [確認執行]  [取消]  [修改]     │
└─────────────────────────────────┘
   │
   ▼
人類回覆 → 繼續 or 取消
```

**LangGraph 把這個叫做 Breakpoint**，是其相比 LangChain 最大的賣點之一。

## 解法成熟度總表

| 解法 | 成熟度 | 建議 |
|------|--------|------|
| 結構化輸出 | ✅ 已標準化 | 所有 Agent 都應該用 |
| Prompt Caching | ✅ API 直接支援 | 馬上開啟，免費省錢 |
| 模型分級 | ✅ 明確有效 | 評估每個任務選對模型 |
| Human-in-the-loop | ✅ 有清楚設計模式 | 定義好哪些動作需確認 |
| Guardrails | ⚠️ 需要客製化 | 根據業務需求設計規則 |
| Verification Step | ⚠️ 增加成本 | 高準確率需求才加 |
| Response Caching | ⚠️ 需要設計快取策略 | 適合 FAQ 類場景 |
