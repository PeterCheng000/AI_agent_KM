# 01 — 演化脈絡：從 LLM 到 Agent

> **核心問題**：LLM 已經很聰明了，為什麼還需要 Agent？它是怎麼一步步演化過來的？

---

## 起點：一個根本矛盾

LLM 很聰明，但它是：
- **被動的** — 要人問，才會答
- **無狀態的** — 每次對話都從零開始
- **沒有工具的** — 只能輸出文字，不能真正「做」任何事

> 這個缺口，催生了 Agent 的概念。

---

## 演化全景

```
靜態 LLM → Chain → ReAct Agent → Tool Use → Multi-Agent
（只能說）  （串流程）（自主決策）  （可靠工具）  （規模協作）
```

每一步都在回答同一個問題：**怎麼讓 AI 在更少人介入的情況下，完成更複雜的任務？**

---

## 第一階段：純語言模型時代（2017–2020）

### 關鍵事件：Transformer 誕生
Google 2017 年論文《Attention is All You Need》，解決了語言的長距離理解問題。

```
傳統 NLP（規則/統計）
    ↓
RNN / LSTM（有記憶，但記憶太短）
    ↓
Transformer（注意力機制，真正理解長距離語意關係）
    ↓
BERT（理解派） / GPT（生成派） — 兩條路線分叉
```

### 這個時期的本質
模型是一個「**極強大的條件機率分佈**」——給定輸入，預測最可能的輸出。

**問題**：它很聰明，但它只會「說」，不會「做」。

---

## 第二階段：讓模型「聽話」（2021–2022）

### 關鍵突破：RLHF

GPT-3 雖然強大，但輸出難以控制。解法是 **RLHF（Reinforcement Learning from Human Feedback）**：

```
預訓練模型
    ↓  人類標記「哪個回答比較好」
SFT（監督微調）
    ↓  訓練 Reward Model
PPO 強化學習
    ↓
InstructGPT → ChatGPT
```

### 關鍵轉折
模型從「文字接龍」變成「**目標導向**」——它開始嘗試達成使用者的意圖，而不只是預測下一個詞。

---

## 第三階段：Chain — 把步驟串起來（2022）

### 核心想法
模型能理解指令後，工程師開始思考：**能不能把多個 LLM 呼叫串在一起，完成複雜任務？**

```
使用者問題
    │
    ▼
[Step 1] 摘要文件
    │
    ▼
[Step 2] 根據摘要提問
    │
    ▼
[Step 3] 整理成報告
```

**LangChain 在這個時期崛起**，核心價值就是讓這種「流程編排」變得簡單。

### 根本限制
流程是人寫死的，**模型只是執行者，不會自己決定要不要做某個步驟**。

---

## 第四階段：ReAct — Agent 真正的思想核心（2023 初）

### 里程碑論文
《ReAct: Synergizing Reasoning and Acting in Language Models》（Yao et al., 2023）

### 核心洞察
讓模型**交替進行「思考」和「行動」**，而不是走固定流程：

```
傳統 Chain：          ReAct Agent：
固定流程              動態決策

Step1→Step2→Step3     Thought→Action→Observation
（人決定步驟）         →Thought→Action→Observation
                      →Final Answer
                      （模型自己決定）
```

### 完整運作範例

```
User: 台積電 2023 年營收比前一年成長還是衰退？

Thought: 我需要查 2023 年和 2022 年的數字，再比較。
Action: search("台積電 2023 全年營收")
Observation: 約 2.16 兆台幣

Thought: 有 2023 了，再查 2022。
Action: search("台積電 2022 全年營收")
Observation: 約 2.26 兆台幣

Thought: 2.16 < 2.26，衰退了。計算幅度。
計算: (2.16 - 2.26) / 2.26 ≈ -4.4%

Final Answer: 2023 年營收約 2.16 兆，較 2022 衰退約 4.4%。
```

模型自己決定要搜尋幾次、搜尋什麼、什麼時候停止。

---

## 第五階段：Tool Use 標準化（2023 中）

### 問題
ReAct 雖然強，但讓模型「猜」要怎麼呼叫工具很不穩定：

```
以前（暴力法）：
模型輸出字串 "Action: search('xxx')"
→ 程式用 regex 解析
→ 格式經常錯誤、容易幻覺
```

### 解法：Function Calling / Tool Use

OpenAI 推出 Function Calling，Anthropic 推出 Tool Use，模型改為輸出**結構化 JSON**：

```json
{
  "tool": "search",
  "arguments": { "query": "台積電 2023 營收" }
}
```

### 意義
讓 Agent 從「研究玩具」變成「可以上線的東西」。工具呼叫變得可靠、可驗證、可型別檢查。

---

## 第六階段：Multi-Agent（2024 至今）

### 為什麼單一 Agent 不夠？

單一 Agent 有三道物理限制：

```
限制一：Context Window 有上限
  500 份合約 → 需要 400k tokens → 直接爆掉

限制二：一個 System Prompt 難以身兼多職
  同時是法律專家 + 資料分析師 + 報告撰寫者 → 角色混亂

限制三：只能串行執行
  100 個任務只能一個一個來 → 速度瓶頸
```

### 解法：分工協作

```
Orchestrator Agent（規劃者）
        │
   ┌────┼────┐
   ▼    ▼    ▼
 Agent A  Agent B  Agent C
（各司其職，可平行執行）
```

### 本質轉變
> **從「個人能力的極限」跨越到「組織協作的規模」。**

---

## 演化本質的一句話總結

| 階段 | 解決什麼 | 留下什麼新問題 |
|------|---------|--------------|
| Chain | 流程太死板 | 模型沒有自主性 |
| ReAct | 不會自主決策 | 工具呼叫不穩定 |
| Tool Use | 工具呼叫不可靠 | 單一 Agent 有物理上限 |
| Multi-Agent | 規模與分工 | 協調複雜、評估困難 |

```
自主程度

Multi-Agent ████████████████████  協作、分工
Tool Use    ████████████████      可靠工具
ReAct       ████████████          邊想邊做
Chain       ████████              照表操課
LLM         ████                  只會回答
                                           → 時間
```
