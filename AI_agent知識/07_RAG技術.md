# 07 — RAG 技術：Agent 怎麼查詢自己的知識庫？

> **核心問題**：LLM 的訓練知識有截止日期，也裝不下所有領域的專業知識。RAG 怎麼讓 Agent 擁有「即時更新的知識庫」？

---

## RAG 是什麼？

**RAG = Retrieval-Augmented Generation（檢索增強生成）**

核心想法：**不要把所有知識塞進模型，而是在需要時去查**。

```
沒有 RAG：
  User: 「我們公司的請假規定是什麼？」
  LLM:  「我不知道你們公司的規定...」（或開始幻覺）

有 RAG：
  User: 「我們公司的請假規定是什麼？」
  系統: 去查詢公司內部文件 → 找到《員工手冊第三章》
  LLM:  「根據員工手冊，特休假計算方式是...」（基於真實文件回答）
```

---

## RAG 的完整流程

### 離線階段（建立知識庫）

```
文件（PDF / Word / 網頁 / ...）
   │
   ▼
① 文件解析（Document Parsing）
  把各種格式轉成純文字

   ▼
② 切割（Chunking）
  把長文件切成適合大小的段落

   ▼
③ 向量化（Embedding）
  把每個 chunk 轉成數字向量

   ▼
④ 儲存（Vector Database）
  把向量和原始文字存入 DB
```

### 線上階段（查詢）

```
用戶問題
   │
   ▼
⑤ 問題向量化
  把問題也轉成向量

   ▼
⑥ 語意搜尋
  在 Vector DB 找最相似的 chunk

   ▼
⑦ 重排（Reranking）
  進一步篩選，確保最相關的排前面

   ▼
⑧ 組合 Prompt
  把搜到的 chunk + 原始問題組合送給 LLM

   ▼
⑨ LLM 生成回答
  根據提供的上下文回答，不亂猜
```

---

## 關鍵步驟詳解

### ② 切割（Chunking）— 最影響效果的步驟

切得不好，再好的搜尋也找不到正確資訊。

**固定字數切割（最笨，但常見）**：
```
每 500 字切一刀：
「...台積電的營收在 202  |  3年達到高峰...」
                    ← 把重要資訊切斷了！
```

**語意切割（Semantic Chunking）**：
```
根據語意完整性切割，確保每個 chunk 是一個完整的概念：
「台積電 2023 年營收達到 xxx，較前年...」← 完整段落
「台積電的主要客戶包括 Apple、NVIDIA...」← 另一個完整概念
```

**階層切割（Hierarchical Chunking）**：
```
Chapter 3: 財務表現（摘要層）
  ├── Section 3.1: 年度營收（中層）
  │     ├── chunk: 2023 年營收數字（細節層）
  │     └── chunk: 季度分布分析（細節層）
  └── Section 3.2: 毛利率分析（中層）
        └── ...

搜尋時：先找章節，再精確定位細節
```

**Overlap（重疊）**：
```
Chunk 1: 第 1~500 字
Chunk 2: 第 400~900 字  ← 和 Chunk 1 重疊 100 字
Chunk 3: 第 800~1300 字 ← 確保邊界資訊不遺失
```

---

### ⑥ 搜尋策略 — Hybrid Search 是現在的標準

**只用向量搜尋（語意搜尋）**：
```
優點：「台北的汽車」能找到「首都的轎車」（理解語意）
缺點：精確關鍵字「型號 ABC-123」可能找不到
```

**只用 BM25（關鍵字搜尋）**：
```
優點：精確匹配關鍵字
缺點：「汽車」找不到「轎車」（不理解語意）
```

**Hybrid Search（業界標準做法）**：
```
向量搜尋結果 ─┐
              ├─ RRF 演算法融合排名 ─→ 最終結果
BM25 結果   ──┘

兩者互補：
- 語意搜尋補捉「意思相近」
- 關鍵字搜尋補捉「精確詞彙」
```

---

### ⑦ Reranking — 精準度的最後一道關卡

```
Hybrid Search 找到 Top 20 個結果
（速度快，但排名不一定準確）
   │
   ▼
Reranker 模型（Cross-encoder）
逐一真正「讀懂」問題和每個 chunk 的關係
   │
   ▼
重新排名，取 Top 3~5
（準確，但需要額外計算）
   │
   ▼
把這 3~5 個 chunk 塞進 Prompt
```

**Reranker 為什麼更準？**

向量搜尋用的是「問題向量」和「文件向量」各自獨立計算相似度。Reranker 是把問題和文件**一起輸入**，直接計算它們之間的關聯性，更準確但更慢。

代表工具：Cohere Rerank、BGE Reranker

## RAG vs Long Context — 2024 最大爭論

Gemini 1.5 Pro 有 100 萬 token context，有人開始問：**RAG 還需要嗎？**

### 目前的業界共識

```
文件少、需要全文理解 → 直接用 Long Context
  例：分析一份完整合約、理解一個完整程式碼庫

文件多、需要即時更新 → 用 RAG
  例：公司內部知識庫（幾千份文件）、新聞資料庫
```

### Long Context 的四個限制

| 面向 | 說明 |
|------|------|
| 費用 | 100 萬 token 費用昂貴，RAG 只付幾個 chunk 的錢 |
| 速度 | 長 context 推理速度慢 |
| 更新 | 文件更新要重新全部送入，RAG 只需更新 Vector DB |
| 注意力 | 「Lost in the Middle」：context 中間的內容容易被忽視 |

### 現在最前沿的做法：兩者結合

```
RAG 找出最相關的 20 個段落
        ↓
把這 20 個段落用 Long Context 一起給模型
        ↓
模型在「小而精準」的 context 裡做推理

→ 又準確又省錢
```

## RAG 效果不好的常見原因

```
問題                    可能原因               解法
──────────────────────────────────────────────────────
找不到相關內容          Chunking 切斷了關鍵資訊  換用 Semantic Chunking
找到了但排名不對         只用向量搜尋             改用 Hybrid Search
找到了但 LLM 沒用到     塞太多 chunk，注意力分散  加 Reranking，只取 Top 3
回答不準確              找到了錯的 chunk          改善文件品質、加 metadata
```

## 常見 Vector Database 比較

| 工具 | 特性 | 適合場景 |
|------|------|---------|
| **Chroma** | 輕量、易上手、本地運行 | 開發測試、小型專案 |
| **Pinecone** | 雲端服務、scalable | 生產環境、大規模 |
| **Weaviate** | 支援 Hybrid Search | 需要關鍵字+語意搜尋 |
| **Qdrant** | 效能優秀、開源 | 自建、效能要求高 |
| **pgvector** | PostgreSQL 擴充 | 已有 PostgreSQL 的系統 |
