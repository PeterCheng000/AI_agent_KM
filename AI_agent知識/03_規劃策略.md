# 03 — 規劃策略：Agent 怎麼「思考」和「決策」？

> **核心問題**：Agent 要完成複雜任務，需要的不只是「回答」，而是「規劃」。有哪些規劃框架，各自適合什麼場景？

---

## 規劃策略的演進邏輯

```
CoT          → 讓模型想清楚再說（推理）
ReAct        → 讓模型邊想邊做（推理 + 行動）
Plan-Execute → 先完整規劃，再分步執行（穩定性）
ToT          → 探索多條路徑，選最好的（創意）
Reflexion    → 執行後反思，不斷修正（準確性）
```

每個策略都是在強化「推理品質」或「執行可靠性」的某一面。

---

## 策略一：Chain of Thought（CoT）

### 核心概念
讓模型把推理過程**逐步寫出來**，而不是直接跳到答案。

```
❌ 沒有 CoT：
User: 小明有 5 顆蘋果，給了 2 顆，又買了 3 顆，還有幾顆？
AI: 6 顆

✅ 有 CoT：
User: 小明有 5 顆蘋果，給了 2 顆，又買了 3 顆，還有幾顆？讓我們一步步來。
AI: 
  Step 1: 原本有 5 顆
  Step 2: 給了 2 顆，剩 5 - 2 = 3 顆
  Step 3: 買了 3 顆，現在有 3 + 3 = 6 顆
  答案：6 顆
```

### 為什麼有效？
模型在生成中間步驟時，後面的 token 能「看到」前面的推理，形成更一致的邏輯鏈。

### 適合場景
- 數學計算
- 邏輯推導
- 多步驟分析
- 任何「直覺容易錯」的任務

### 觸發方式
只需在 prompt 加上：
- 「讓我們一步步思考」
- 「請先列出推理過程」
- 「Think step by step」

---

## 策略二：ReAct（Reasoning + Acting）

### 核心概念
不只推理，還要**交替行動與觀察**。每次行動後，把結果納入下一步的思考。

```
Thought:  我需要知道台積電現在的股價
Action:   search("台積電 TSMC 股價")
Observation: TSMC 目前股價為 xxxxxx

Thought:  我有股價了，現在需要本益比，要查 EPS
Action:   search("台積電 2024 EPS")
Observation: 2024 年 EPS 約 xx 元

Thought:  本益比 = 股價 / EPS = xxx，可以回答了
Final Answer: 台積電目前本益比約 xx 倍
```

### 與 CoT 的差異

| | CoT | ReAct |
|--|-----|-------|
| 有無工具 | 無，純推理 | 有，可呼叫外部工具 |
| 資訊來源 | 模型本身知識 | 即時查詢 |
| 適合任務 | 需要推理的問題 | 需要最新資訊的問題 |

### 適合場景
- 需要查詢即時資訊
- 需要計算或執行程式碼
- 需要多工具協作

---

## 策略三：Plan-and-Execute

### 核心概念
把規劃和執行**明確分成兩個階段**。先生成完整計劃，再逐步執行。

```
Phase 1 — 規劃（只輸出計劃，不執行）：
  [1] 搜尋台積電近五年財報數據
  [2] 搜尋聯電近五年財報數據
  [3] 計算各項財務指標
  [4] 比較並分析差異
  [5] 生成報告

Phase 2 — 執行（按計劃逐步做）：
  執行步驟 [1]... 完成
  執行步驟 [2]... 完成
  執行步驟 [3]... 完成
  ...
```

### 為什麼比 ReAct 更穩定？

ReAct 的問題：每一步都在「邊想邊做」，容易在執行途中迷失目標，或把小的中間結果誤認為最終答案。

Plan-and-Execute：先有全局視野，執行時不容易偏離。

### 適合場景
- 步驟多、任務長（超過 5 個步驟）
- 任務結構相對清晰（例如：寫報告、做分析）
- 需要人類審查計劃再執行

### 缺點
- 如果執行過程中發現計劃有誤，需要重新規劃
- 不如 ReAct 靈活

---

## 策略四：Tree of Thought（ToT）

### 核心概念
不走單一路徑，而是**同時探索多條可能性**，再選最優解。

```
            問題
           /    \
        路徑A   路徑B
        /  \       \
      A1   A2      B1
                  /   \
                B1a   B1b  ← 評估：B1b 最好
```

### 運作方式

```
Step 1: 生成多個可能的第一步（廣度）
Step 2: 對每個選項評估「這個方向有希望嗎？」
Step 3: 選擇最有希望的方向繼續展開
Step 4: 如果走死路，回退（Backtrack）換另一條路
Step 5: 找到最佳解
```

### 適合場景
- 需要創意發想（有多種可能解法）
- 問題有明確的評估標準
- 例：寫一個能通過測試的程式碼、設計一個策略

### 代價
Token 消耗大（探索多條路徑），速度慢，成本高。

---

## 策略五：Reflexion（反思修正）

### 核心概念
Agent 執行後，**自我評估結果的品質**，如果不夠好就修正再試。

```
第一次執行
    ↓
得到結果
    ↓
自我反思：「這個答案正確嗎？有沒有遺漏？」
    ↓
┌───Yes────────No───┐
輸出              重新嘗試
               （帶著反思記憶）
                    ↓
              第二次執行（更好）
```

### 關鍵設計：反思記憶

Reflexion 會把「上次哪裡錯了」存進記憶，讓下次執行能避免同樣錯誤：

```
反思記憶：
- 上次用 search("TSMC") 沒找到中文資料，
  下次應該用 search("台積電") 再試一次
- 上次計算忽略了四捨五入，
  下次需要明確指定精度
```

### 適合場景
- 高準確率要求的任務（程式碼生成、事實查詢）
- 允許多次嘗試的場景
- 任務有明確的「成功標準」可以自動評估

---

## 策略比較總表

| 策略 | 工具使用 | 自我修正 | Token 消耗 | 適合任務複雜度 |
|------|---------|---------|-----------|--------------|
| CoT | ❌ | ❌ | 低 | 中（推理） |
| ReAct | ✅ | 部分 | 中 | 中高（即時資訊）|
| Plan-Execute | ✅ | 部分 | 中 | 高（多步驟）|
| ToT | ✅ | ✅ | 高 | 高（創意探索）|
| Reflexion | ✅ | ✅ | 中高 | 高（高準確率）|

---

## 怎麼選擇規劃策略？

```
任務需要最新資訊？
  → Yes → ReAct 或 Plan-Execute

任務超過 5 個步驟？
  → Yes → Plan-Execute

任務需要高準確率，錯了要重試？
  → Yes → Reflexion

任務需要探索多個可能解法？
  → Yes → Tree of Thought

只是需要深度推理，不需要工具？
  → CoT 就夠了
```
