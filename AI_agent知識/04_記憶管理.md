# 04 — 記憶管理：Agent 怎麼「記住」東西？

> **核心問題**：LLM 本身沒有記憶，每次對話都從零開始。怎麼讓 Agent 記住短期對話、長期知識、和用戶偏好？

---

## 為什麼記憶很重要？

沒有記憶的 Agent 就像：
- 每次打電話給客服，對方都不知道你是誰
- 每次跟同事開會，對方都不記得上次的決定
- 每次使用工具，工具都不知道你的偏好

**記憶讓 Agent 從「問答機器」變成「真正的助手」。**

---

## 記憶的四個層次

```
速度  ◄─────────────────────────────────► 容量

In-Context  →  Summary  →  Entity  →  Vector DB
（最快）                              （最大）
```

### Layer 1：In-Context Memory

**原理**：直接把資訊放在 context window 裡。

```
┌──────────────────────────────────┐
│  System Prompt                   │
│  [對話歷史 - 最近 10 輪]         │  ← 直接在這裡
│  當前問題                        │
└──────────────────────────────────┘
```

**優點**：速度最快，模型直接「看到」
**缺點**：Context 有上限（128k tokens 大約 10 萬中文字），超過就要截斷

**適合**：當前對話的近期上下文

---

### Layer 2：Conversation Summary Memory

**原理**：舊對話不直接保留，用 LLM 自動壓縮成摘要。

```
原始對話（2000 tokens）：
  User: 我在找一個 Python 框架做 web scraping
  AI: 推薦 Scrapy 或 BeautifulSoup...
  User: Scrapy 和 BeautifulSoup 有什麼差別？
  AI: Scrapy 是完整框架... BeautifulSoup 是解析庫...
  User: 好，我選 Scrapy
  AI: 好的，以下是安裝步驟...

壓縮後摘要（100 tokens）：
  「用戶選擇使用 Scrapy 做 web scraping，
   已討論過與 BeautifulSoup 的比較。」
```

**優點**：大幅節省 token，長對話也能維持上下文
**缺點**：細節會損失，摘要品質依賴 LLM

**適合**：多輪長對話

---

### Layer 3：Entity Memory

**原理**：從對話中抽取重要實體，以結構化格式存儲。

```python
# 對話過程中自動抽取
entity_store = {
    "用戶": {
        "名字": "Alice",
        "職業": "後端工程師",
        "程式語言": ["Python", "Go"],
        "偏好": "簡潔的回答",
        "目前任務": "建立 web scraping 系統"
    },
    "專案": {
        "名稱": "競品資料收集器",
        "技術棧": ["Scrapy", "PostgreSQL"],
        "狀態": "開發中"
    }
}
```

**優點**：精確，可直接查詢特定事實，不受 context 限制
**缺點**：需要明確定義要追蹤哪些實體，靈活性較低

**適合**：需要記住用戶資料、專案狀態、特定偏好

---

### Layer 4：Vector Memory

**原理**：把重要對話片段向量化，存入 Vector Database，用語意搜尋找相關記憶。

```
對話片段 → Embedding 模型 → 向量 → 存入 Vector DB

查詢時：
當前問題 → Embedding → 搜尋最相關的歷史片段 → 載入 Context
```

```
示意圖：
                Vector DB
               ┌───────────────────────────────┐
               │  [3個月前] 用戶問過 Scrapy 設定│
               │  [1個月前] 討論過資料庫 schema │
               │  [上週] 解決了爬蟲被封 IP 問題│
               │  [昨天] 用戶說要加代理池功能  │
               └───────────────────────────────┘
                          ↑
  當前問題：「我的爬蟲又被封了」
  → 語意搜尋 → 找到「上週解決爬蟲被封 IP 問題」
  → 載入該段記憶到 Context
```

**優點**：容量無限，語意搜尋很聰明，可跨對話
**缺點**：有延遲（要做搜尋），可能找到不相關的記憶

**適合**：長期知識、跨對話記憶、大型知識庫

---

## 四層記憶的協作

實際系統通常**同時使用多層記憶**：

```
用戶發問
   │
   ▼
[Layer 4] Vector DB 搜尋相關歷史記憶
   │
   ▼
[Layer 3] 載入 Entity Memory（用戶資料、當前狀態）
   │
   ▼
[Layer 2] 載入對話摘要（最近幾次對話的摘要）
   │
   ▼
[Layer 1] 最近 N 輪完整對話
   │
   ▼
組合成完整 Context → LLM 回答
```

## 記憶的寫入時機

不是所有資訊都需要記住。什麼時候寫入？

```
❌ 不需要記：
  「今天天氣怎麼樣？」→ 臨時查詢，不值得記

✅ 需要記：
  「我叫 Alice，是後端工程師」 → Entity Memory
  「我偏好簡潔的回答」        → Entity Memory
  「我們上次決定用 PostgreSQL」→ Summary / Vector
  「解決爬蟲被封的方案」      → Vector（可能再用到）
```

### 觸發記憶寫入的規則（常見設計）

```python
# 簡單規則
if "我叫" in message or "我是" in message:
    → 寫入 Entity Memory

if 對話輪數 > 10:
    → 壓縮前 5 輪到 Summary Memory

if 本次對話包含「解決方案」或「重要決定」:
    → 寫入 Vector Memory
```

## 現在還沒解決的問題

### 1. 記憶的取捨
- 什麼值得記？什麼要忘記？
- 目前多半靠人工規則，沒有優雅的通用解

### 2. 記憶的一致性
```
3個月前 Alice 說：「我用 Python」
1個月前 Alice 說：「我最近在學 Go」
現在問：「Alice 用什麼語言？」

→ 怎麼處理矛盾的記憶？哪個更新？
```

### 3. 記憶的隱私
- 多用戶系統怎麼確保記憶完全隔離？
- 記憶要存多久？怎麼讓用戶刪除自己的記憶？

---

## 記憶 vs RAG 的差異

這兩個概念常被混淆：

| | 記憶（Memory） | RAG |
|--|--------------|-----|
| 內容來源 | 對話歷史 | 外部文件/知識庫 |
| 目的 | 記住用戶、記住上下文 | 查詢特定知識 |
| 更新方式 | 每次對話自動更新 | 手動更新文件 |
| 典型用途 | 個人化助手 | 企業知識庫問答 |

兩者常常同時使用，共同組成 Agent 的完整知識系統。
