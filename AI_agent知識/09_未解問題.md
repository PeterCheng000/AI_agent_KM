## 未解問題一：Memory 一致性

### 問題的本質

Agent 記憶越來越多，但記憶之間可能出現矛盾：

```
3 個月前記憶：「用戶 Alice 使用 Python 開發」
1 個月前記憶：「Alice 說她最近在全力學 Go」
上週記憶：「Alice 問了一個 Python 的問題」

現在問：「Alice 主要用什麼語言？」
→ 哪個記憶是對的？哪個更新？
```

### 目前的處理方式（都是打補丁）

**方式A：時間戳優先**
```
最新的記憶優先 → 但 Alice 可能同時用兩種語言
```

**方式B：Confidence Score**
```
每條記憶附上置信度，低置信度的被高置信度覆蓋
→ 但誰來評置信度？模型自己評也可能錯
```

**方式C：詢問用戶確認**
```
「我注意到你之前說用 Python，但最近又提到 Go，
你現在主要用哪個？」
→ 準確，但用戶體驗差（感覺 AI 很笨）
```

### 為什麼難解？

這本質上是個「知識更新」問題：

```
我怎麼知道一條新資訊是：
  a) 舊資訊的更新（要覆蓋）
  b) 與舊資訊並存的新事實（要追加）
  c) 暫時的例外（不用改）

沒有明確的上下文，這很難判斷。
```

### 目前最前沿的方向

- **Temporal Knowledge Graph**：用時間線組織記憶，每條事實都標記「有效期間」
- **Episodic Memory**：像人類一樣，把記憶分成「事件」而不是「事實」，保留完整脈絡

**現狀**：都還在研究階段，沒有可以直接套用的成熟方案。

## 未解問題二：Multi-Agent 最佳協作模式

### 問題的本質

```
同一個任務，可以用不同的 Multi-Agent 架構：
  方案A：1 個強 Orchestrator + 多個 Worker
  方案B：多個對等 Agent 互相討論
  方案C：流水線（A → B → C → D）
  方案D：混合架構

哪個最好？什麼情況下用什麼？沒有定論。
```

### 當前爭論的核心問題

**Agent 應該「共享狀態」還是「傳遞訊息」？**

```
共享狀態（Shared State）：
  優點：所有 Agent 都能看到全局，溝通簡單
  缺點：可能有並發衝突，狀態越來越複雜

訊息傳遞（Message Passing）：
  優點：責任邊界清楚，適合分散式
  缺點：訊息格式設計困難，容易資訊孤島
```

目前 LangGraph 用共享狀態，AutoGen 用訊息傳遞——兩個主流框架各走一條路，尚無定論。

**Agent 的「自主程度」要多高？**

```
高自主：Agent 自己決定要不要找其他 Agent 幫忙
  優點：靈活
  缺點：難以預測、難以調試

低自主：Orchestrator 完全控制每個 Agent 的行為
  優點：可預測、可控
  缺點：靈活性差，Orchestrator 成為瓶頸
```

### 現狀

每個框架（LangGraph、AutoGen、CrewAI）都有自己的一套理念，業界還沒有像「RAG 架構」那樣廣泛認可的最佳實踐。

**目前最保守的建議**：
> 從最簡單的 Orchestrator-Worker 開始，等遇到具體瓶頸再調整。不要過度設計。

## 未解問題三：Agent Evaluation（評估方法論）

### 為什麼這是最難的問題？

傳統軟體測試：
```
輸入 → 確定性輸出 → 對比預期 → Pass/Fail
```

Agent 測試：
```
輸入 → 不確定性輸出（每次可能不同）
     → 過程也不固定（可能呼叫不同工具，不同次數）
     → 怎麼定義「正確」？怎麼知道「夠好」？
```

### 三個核心難題

**難題A：最終結果的評估**
```
「幫我寫一份市場分析報告」
→ 報告寫出來了，但怎麼判斷好不好？
→ 字數夠？格式正確？邏輯清晰？數據準確？
→ 每個標準都有主觀成分
```

**難題B：過程的評估**
```
Agent 用了 10 步完成任務
另一個 Agent 用了 3 步完成同樣任務

誰比較好？
→ 步驟少的不一定準確
→ 步驟多的不一定更可靠
→ 過程本身很難量化評估
```

**難題C：邊界案例的覆蓋**
```
傳統測試：可以列舉所有輸入組合
Agent 測試：輸入空間無限大，無法窮舉

用戶可能問任何問題、以任何方式問
→ 怎麼確保 Agent 在「從未見過的情況」也能正常運作？
```

### 目前主流的做法（都有缺陷）

**做法一：LLM-as-Judge**

用另一個 LLM 來評估輸出品質：
```
評審 Prompt：
「以下是一個 AI 助理的回答：[回答內容]
請從以下維度評分（1-5分）：
- 準確性
- 完整性  
- 格式清晰度
並說明理由。」
```

缺陷：
- 評審本身可能有偏見（「GPT-4 偏好 GPT-4 風格的回答」）
- 評審也可能幻覺
- 評審結果不穩定（同樣答案，多次評審可能給不同分數）

**做法二：黃金資料集（Golden Dataset）**

人工標注一批「標準答案」，測試 Agent 能不能達到：
```
問題 1：...  → 標準答案：...  → Agent 回答：... → 比對相似度
問題 2：...  → 標準答案：...  → Agent 回答：... → 比對相似度
```

缺陷：
- 標注成本高（需要領域專家）
- 資料集有限，不能覆蓋真實世界的多樣性
- Agent 可能「過擬合」測試集

**做法三：Trajectory Evaluation（過程評估）**

不只看最終答案，也評估 Agent 的推理步驟：
```
評估項目：
  ✓ 是否呼叫了正確的工具？
  ✓ 工具呼叫的參數是否合理？
  ✓ 觀察到結果後，推理是否正確？
  ✓ 是否在適當的時候停止？
```

這是目前最有潛力的方向，但還在研究階段。

### 代表工具

| 工具 | 特性 |
|------|------|
| **LangSmith** | LangChain 官方，追蹤每一步執行過程 |
| **Braintrust** | 強調 LLM-as-Judge 的 Evaluation 流程 |
| **Ragas** | 專門評估 RAG 系統品質 |
| **PromptFoo** | 開源，支援各種評估策略 |

## 新興議題：值得關注但還太早下結論

### Long Context 是否會淘汰 RAG？

```
現在：
  Long Context 有物理和成本限制，RAG 仍然必要

趨勢：
  Context 越來越長（1M → 10M token？）
  成本越來越低
  → 某些 RAG 場景可能真的被取代

但：
  RAG 的知識更新能力（不用重新訓練）仍是優勢
  → 可能演化成「混合型」，而不是誰淘汰誰
```

### Agent 是否會取代傳統軟體？

```
某些場景（非結構化任務、需要推理的流程）：Agent 明顯更好
某些場景（高確定性、高效能需求）：傳統程式碼更好

未來可能：
  Agent 負責「判斷」，傳統程式碼負責「執行」
  兩者協作，而不是取代
```
